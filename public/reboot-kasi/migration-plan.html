Ubuntu Town OS

Source-of-Truth Migration Plan · from GitHub audit

One truth: consolidate onto Supabase.

The audit found the real blocker: Ubuntu Town runs on two disconnected databases. The plan is to make the Supabase Postgres project the single source of truth — for the web, the coordinator portal, and the field PWA.

Repos: ubuntu-town-web · ubuntu-town-field-osOwner: Kekelebaka5 July 2026

01 — The finding

Two databases, not one

"Make Supabase the source of truth 100%" turned out to be bigger than a config tweak. The two repos run on two separate databases that don't talk to each other, and the operational heart of the system is currently on the wrong one for the goal.

Cloudflare D1  field OS

ubuntu-town-field-os · db "ubuntu-town-db"
  • Holds the richest data. Full coordinator lifecycle (applications → reviews → coordinators → assignments → onboarding → mission events → reliability → escalations), coordinator_proofs (R2-backed), town_heartbeat_scores, mobile_town_signals, mobile_opportunity_interests, cases.
  • Sub-products in the same DB: Family House (fh_*), KasiBuy (kb_*); a separate Jeff bot DB (POPIA data-rights tables).
  • SQLite — no RLS, no enums, one FK. App-enforced integrity.

Supabase Postgres  web

ubuntu-town-web · @supabase/ssr · 14 tables
  • provinces, towns, users, profiles, coordinators (earnings = a raw JSON blob, no ledger), town_signals, town_metrics, opportunities, businesses, services, access_points, events, stories, partner_offers, cv_profiles.
  • The public site is Supabase-native (SSR clients + typed schema) — but the authenticated (app-pages) shell is still starter boilerplate.
  • The proper place for the single source of truth — once the field-OS data lands here.
Blocker · must clear first

The Supabase project ubuntu-town-os (afiokbhuxfdacbsipoqk) is PAUSED (INACTIVE) — it timed out on connect. A paused database cannot be a source of truth. Also: the project ref is not committed anywhere in the repos (referenced only via the SUPABASE_PROJECT_REF build env var) — confirm this is the right project before migrating.

Security · surfaced by the audit

A legacy Worker ubuntuacademy.chiefops26.workers.dev still exposes a broad unauthenticated /api/* surface (signals, proof, coordinators, towns, governance/roles…). It must be secured or retired as part of this migration.

04 — The phased plan

Seven phases — start the moment Supabase is resumed

0
Unblock · day 1

Resume & confirm the project

Resume ubuntu-town-os, disable auto-pause (paid tier / keep-warm), confirm the ref matches SUPABASE_PROJECT_REF, set env vars. Then I inspect the live schema + RLS.

1
Inventory · freeze

Export & snapshot D1

Dump all D1 tables (field-os) + the Mailchimp pack. Freeze writes to the legacy Worker. Produce the definitive field-inventory.

2
Model

Extend the Supabase schema

Migrations for the coordinator lifecycle, proofs, opportunity_points, workpack_instances, unified town_opportunity_score, the payout_runs/lines ledger, and roles + RLS on every table.

3
ETL

Migrate D1 → Supabase + import applicants

Scripted, idempotent ETL (D1 SQLite → Postgres), media to Supabase Storage/R2. Import the 46 applicants + 34 towns into the coordinator + town tables. Reconcile duplicates.

4
Repoint

Wire the three clients to Supabase

Web already Supabase-native — build out the (app-pages) coordinator portal. Wire the field PWA's offline queue to real Supabase writes (replace the "mark synced" stub). Kopano/workers use the service role.

5
Retire

Decommission D1 + the open Worker

Secure or shut the unauthenticated Worker API; demote D1 to edge cache or retire it. One write path: Supabase.

6
Verify

Cutover & parity checks

Row-count + spot-check parity, RLS tests, the mock-fallback disabled in prod, Lighthouse/PWA green. Sign-off = the definition of done below.

05 — "Integrate everything from the old one"

Feature-parity checklist

  • Coordinator applications, reviews, onboarding, reliability, escalations — migrated & visible in the portal
  • Signals feed (mobile + web) — one reconciled table, powering the field PWA capture
  • Proof capture with media (R2/Storage) + human review (Kopano recommends, human approves)
  • Town Opportunity Score — one unified, persisted model (replaces the three competing ones)
  • Opportunity Points (converted spazas) + workpack instances linked signal→workpack→proof→memory
  • Revenue-share payouts — a real ledger, feeding the coordinator economics model
  • Roles & governance — real table + RLS gating the coordinator portal
  • Family House & KasiBuy — migrated in phase 2 (kept, not lost)

06 — Guardrails

Do not skip these

  • RLS on every table before any real coordinator data lands — the generated types show no policies; assume none exist yet.
  • Kill the unauthenticated Worker API — it currently exposes coordinators, proof, governance/roles publicly.
  • POPIA — the Jeff data-rights tables show the intent; carry consent + data-rights into the unified schema.
  • Disable the mock-data fallback in production (web silently serves mock data when creds are missing).
  • Turn off typescript.ignoreBuildErrors on the web build before cutover.
  • Secrets stay in Cloudflare/Supabase — never commit the project ref or service-role key (per the repo's own AGENTS.md).
  • No deploy without owner approval — AGENTS.md rule; this plan is executed on your go, not autonomously.

07 — Definition of done

What "Supabase 100%" means

  • Every client (web, portal, PWA, workers) reads/writes only Supabase
  • D1 holds no unique source data; the legacy Worker is secured or gone
  • RLS enforced; no public data leaks; mock fallback off in prod
  • The 46 applicants + 34 towns + all field-OS ops data live in Supabase
  • Payout ledger + unified town score + workpack instances exist and are populated
  • Parity verified (row counts + spot checks); rollback snapshot retained

Signals become workpacks. Workpacks become proof. Proof becomes memory — in one place.
